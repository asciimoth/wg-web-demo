HTTP over VTun over WireGuard over SOCKS-over-WebSocket

This browser node uses socksgo for SOCKS-over-WebSocket, batchudp + wgo for the WireGuard transport, and gonnect-netstack/vtun for the userspace IP stack. It is designed for one WireGuard node per tab.

browser WASM node
one WG peer per tab
works with external WG peers
tab-to-tab pairing supported
extensive in-page logging

Typical test flow: 1. start a socks-over-websocket server such as gost -L "socks5+ws://:1080?udp=true&udpBufferSize=4096&bind=true", 2. open one or two tabs with this demo, 3. keep the default proxy URL or use socks5+ws://127.0.0.1:1080?bind=true&gost=true, 4. start each node, 5. exchange each tab’s public key and advertised endpoint, 6. set the peer allowed IPs, 7. click Start / Apply Node again or let the next server/request action auto-apply the pending peer config, 8. start the in-tunnel HTTP server on one side and request it from the other side.

Peer Config

For two tabs, each side should advertise a different tunnel IP and include the remote tab’s tunnel IP in Peer allowed IPs. Changing Local tunnel IP auto-updates the default peer IP and request URL until you override them manually. If you edit these fields while the node is already running, the new config is applied on the next Start / Apply Node, server start, or request.

Share / Import

Copying exports the current node config plus live runtime details such as the public key, advertised endpoint, and active in-tunnel server URL. Importing maps that data into this tab’s peer config, flips the local tunnel IP to the complementary address when possible, and can auto-run a request to the remote server.
